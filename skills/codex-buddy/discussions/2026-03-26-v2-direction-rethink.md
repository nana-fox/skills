# 讨论：v2.0 方向重塑 — 从 Mode A/B/C 到对话协议

**日期：** 2026-03-26 | **模式：** Mode B → Mode C（四轮） | **结果：** 用户决定走方向 B（重构）

---

## 话题

用户反馈：优化应从 触发、使用、格式、交流、反馈 五个维度考虑。
总目标：让 AI 交流起来，消除幻觉，且机制合理。
具体问题：Mode A/B/C 太死板、交流不是双向的、反馈没有落地。

---

## 第一轮：项目方向评估

**Claude：** 最近 4 版趋势是规范层越来越厚，skill 从"工具"变成"制度"。核心动作（调 Codex、对比结论）没有变得更容易。

**Codex：** 方向没偏，但到了拐点。项目很会定义"应该怎么做"，但还不够会证明"真实发生了什么"。建议停加规则，先补可回放证据。

---

## 第二轮：用户引导方向

**用户反馈原文：**
- 交流应是双向的：Codex 有疑问应该能追问 Claude，不是只有 Claude 问 Codex
- 反馈两层：(1) 另一个 AI 的回答帮自己减少幻觉，有价值的存记忆 (2) 项目从使用反馈中迭代
- Mode A/B/C 不够灵活，"总感觉怪怪的"

**Claude 独立分析：**
> Mode 应该自然涌现，不该预选。当前分类标准是"信息污染程度"——技术参数，不是用户的自然思路。

**Codex 独立方案：**
> 把 Mode 改成 5 个协议动作（Probe/Ask/Challenge/Test/Synthesize），按需组合，不互斥。
> 把"追问"定义为一等输出（Codex 可返回 need_info）。
> 反馈分两层：会话 incident → 稳定记忆（晋升条件：重复2次/代价高/已转eval/跨项目）。

---

## 第三轮：Claude 质疑 → Codex 自我修正

**Claude 的 5 个质疑：**
1. 5 动作可能比 3 模式更复杂
2. Codex 不会稳定遵守回复信封
3. Ask 默认触发会翻倍调用成本
4. VEM 降级后触发问题谁解决
5. 最小可行版本是什么

**Codex 自我修正：**
- 5 动作降为内部执行动词，不替代 Mode A/B/C
- 回复信封放 Claude 侧（弱约束+强适配）
- Ask 改为按需升级，默认单次 Probe
- **收回 VEM 降级建议**，VEM 必须保留为外层闸门
- 引用 CLAUDE.md 约束论证保留 Mode A/B/C

**关键张力：** Codex 用项目内约束（"Mode A/B/C 不可破坏"）论证保守路线，但用户作为项目所有者质疑这条约束本身。

---

## 用户决策

**方向 B：放松 CLAUDE.md 约束，用更自然的对话协议替代 Mode A/B/C。**

---

## 收敛的设计原则

来自三轮讨论的共识（Claude + Codex + 用户）：

1. **VEM 保留为外层闸门**（什么时候调 Codex）
2. **Mode 不再预选**，改为自然对话流程（Probe → 按需 Ask/Challenge → Synthesize）
3. **默认单次 Probe**，Ask 只在明确触发条件下升级
4. **Claude 侧解析**（弱约束+强适配），不要求 Codex 遵守固定格式
5. **Evidence Packaging Rule 保留**
6. **双向交流**：Codex 可以表达不确定/追问，Claude 通过 resume 回应
7. **反馈两层**：会话 incident → 记忆晋升（有条件门槛）
8. **协议必须对真实 Claude 可执行**，< 150 行

## 独立性验证

- [x] 三轮中 Codex 均在 Claude 给出方案后独立分析
- **Codex 发现了 Claude 没有提出的什么：**
  - 5 动作的抽象层级混淆问题（Challenge 是交互，Test 是验证，不是同一层）
  - "弱约束+强适配" vs "强约束+脆解析"
  - 第二轮自我修正：收回 VEM 降级、收回 5 动作替代 Mode
  - Ask 的 3 个具体触发条件
- **Claude 因为 Codex 的输出改变了什么：**
  - 认同 5 动作不应成为新的显式模式系统
  - 采纳 Claude 侧解析（不依赖 Codex 结构化输出）
  - 认同 Ask 按需而非默认
- **本轮对话是否真实有效：** 高度有效 — Codex 在第二轮自我修正了第一轮的过度设计，Claude 的质疑是触发修正的关键
