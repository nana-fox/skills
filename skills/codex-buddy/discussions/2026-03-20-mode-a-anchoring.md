# 讨论：Mode A 锚定风险

**日期：** 2026-03-20 | **模式：** Mode A（Claude 先独立方案，Codex 独立审查） | **结果：** 快速收敛，Codex 额外发现"证据打包"问题

---

## 话题

v1.7 中 Codex 指出：Mode A 作为默认入口天然带锚定风险。即使写了"先独立判断再对比"，只要 Claude 的结论已经进 prompt，Codex 的独立性就会被削弱，skill 容易运行成"受控的复述与补充"。本轮主题是重新定义 Mode A 的适用边界，把 Mode B 提升为默认独立模式。

---

## 第一轮：各自开场

**Claude：**

> 当前问题：Mode A 的设计要求 Codex"先独立判断，再与 Claude 对比"，但 Claude 的结论/假设已经在 prompt 里——这是内在矛盾。即使字段隔离，锚定效应仍然存在。V2/V3 场景下这种"看起来独立、其实被锚定"的错觉验证尤其危险。
>
> 改动方案：
> 1. 在 Mode A 的"适用"行里加 `(V0/V1 适用)`，同时加一句警告：V2/V3 场景 Mode A 存在锚定风险，建议改用 Mode B
> 2. 升级链路说明里，V2/V3 场景下默认从 Mode B 开始
>
> 这样改动集中，不破坏现有结构，只是让适用条件更清晰。

**Codex（原始输出）：**

> 我建议这轮不要只微调措辞，而是把 `Mode A` 从"默认入口"降级成"受限模式"。否则文档会继续一边提醒锚定风险，一边在操作层面鼓励锚定。
>
> **改动前：**
> ```md
> 先用 Mode A，发现分歧来自方案空间时升 B，仍有不可解冲突时升 C。
> ```
>
> **改动后：**
> ```md
> 默认不先选 Mode A。先按验证矩阵确定验证级别，再按任务性质选 Mode：
> - 需要独立判断来影响执行决策：先用 Mode B
> - 已有 Claude 成型结论，只做定向审查或找漏项：可用 Mode A
> - 已出现可检验的核心分歧：升 Mode C
> - V2/V3 场景：先决定验证动作；模型协作只能服务于验证，不得替代验证
> ```
>
> Mode A 改为"受限使用，非默认"，加入明确的不适用条件：
> - 需要决定"是否执行"的 V2/V3 场景
> - 架构/选型等存在多可行方案、需要真正独立展开方案空间的场景
> - Claude 尚未形成稳定结论，只是想让 Codex 帮忙"再想一遍"
>
> Mode B 提升为"默认独立模式"，优先用于：
> - 会影响执行/回滚/升级路径的判断
> - 架构取舍、技术选型、复杂 root-cause 分析
> - V1/V2 中需要避免 Claude 先验结论污染的场景
>
> 在 Prompt 独立性协议里加第七条硬规则 Mode-A Boundary：
> - 只有当任务目标是"审查 Claude 现有结论"而不是"获得独立判断"时，才允许使用 Mode A
> - 只要本轮输出将直接影响执行/no-execute 决策，默认先用 Mode B 或直接做外部验证
> - V2/V3 不得以 Mode A 作为首轮协作模式
>
> 建议新增 Mode 选择器决策表：
> | 问题 | 是 | 否 |
> |---|---|---|
> | 这次需要真正独立的一审判断吗？ | Mode B | 继续下一问 |
> | Claude 是否已有成型结论，且本次只做定向审查？ | Mode A | 继续下一问 |
> | 双方是否已出现可检验核心分歧？ | Mode C | 不必调用 Codex |
> | 是否属于 V2/V3 且会影响执行？ | 先定验证动作；B 可辅助，A 不作首轮 | 常规按上表 |
>
> 引入新问题：
> - 成本和时延会上升（Mode B 比 Mode A 更贵）
> - 使用门槛变高（需要先判断"独立判断 vs 定向审查"）
> - 综合责任更重（Mode B 产出两套独立答案后，Claude 需要认真做 synthesis）
> - 某些轻量场景会显得过严
>
> **我最担心的设计缺陷（超出本轮主题）：**
>
> 我最担心的不是 Mode A 本身，而是这个 skill 把"模型独立性"和"证据选择权"混为一谈。
>
> 文档现在非常重视"不要把 Claude 的结论传给 Codex"，但对"Claude 先筛选哪些证据给 Codex"约束不够。现实里最常见的失败不是明显把答案塞进 prompt，而是 Claude 先把证据裁成一版叙事，Codex 只能在这个被筛过的证据盒子里独立思考。这样即便用了 Mode B，异质性也还是假的。
>
> 如果只允许我优先修一个结构缺陷，我会加一条 Evidence Packaging Rule：
> ```md
> 默认优先传"原始可观察证据"而不是 Claude 总结：
> - 代码片段优先于代码解释
> - 原始报错/日志优先于错误归因
> - 命令输出优先于结论摘要
>
> 若必须做证据筛选，必须显式列出：
> - 为什么选这些证据
> - 哪些相邻证据未包含
> ```
>
> 这比单纯把 Mode A 改成非默认更关键，因为它解决的是所有 Mode 都会遇到的上游污染问题。

## 对 SKILL.md 的改动（v1.8）

1. **工作模式入口**：升级链路改为 Mode 选择器决策表（4行），明确各场景选哪个 Mode
2. **Mode A 标题**：`Review（默认）` → `Review（受限，非默认）`，加禁用场景说明
3. **Mode B 标题**：`Parallel（独立并行）` → `Parallel（默认独立模式）`，加优先使用场景
4. **七条硬规则**：新增 `Mode-A Boundary`（任务目标是获得独立判断时禁用 Mode A；V2/V3 禁用）
5. **Verification Matrix**：末尾加 Mode 约束（V2/V3 与 Mode 选择的绑定规则）

SKILL.md 行数：169 → 177 行（+8 行）

## Codex 超出本轮主题的发现（纳入下轮 Agenda 首位）

> 这个 skill 把"模型独立性"和"证据选择权"混为一谈。Claude 先筛选哪些证据给 Codex，这本身就是锚定。即便用了 Mode B，如果 Claude 传的是"处理过的证据叙事"而不是原始可观察事实，异质性还是假的。需要 Evidence Packaging Rule：代码片段 > 代码解释，原始报错 > 错误归因，命令输出 > 结论摘要。

**Claude 的判断：** Codex 说得准确。模型独立性（不传结论）和证据独立性（不传筛选过的叙事）是两个不同的问题，当前 skill 只解决了前者。Evidence Packaging Rule 影响所有 Mode，是比 Mode A 降级更根本的修复。本轮先落 Mode 选择逻辑，下轮专做证据打包规则。
