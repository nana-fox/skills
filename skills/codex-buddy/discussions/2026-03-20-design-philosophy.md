## 问题

从软件设计哲学、AI 协作模式、认知偏差防范等角度独立评述 codex-buddy 的核心理念和设计权衡：
1. 这种设计解决了什么根本问题
2. 三种模式的适用边界如何划分
3. 有哪些潜在的设计缺陷或改进空间

---

## Claude 独立视角

### 核心问题：AI 的"顺畅陷阱"

单一模型最危险的失效模式不是"不知道"，而是**系统性地、流畅地把错误合理化**。RLHF 训练天然鼓励模型生成"听起来对"的答案，而不是"实际上对"的答案。codex-buddy 的本质是：在置信度最高的时刻，强制引入异质性对抗。

触发条件里 **"回答异常流畅时"** 这条最有洞察力——它捕捉到了 AI 特有的认知危险信号：越流畅越可能是在编故事。

### 三种模式的分层逻辑

| 模式 | 本质 | 适用边界 |
|------|------|---------|
| Mode A (Review) | 产物审计 | 已有结论，怀疑执行层错误 |
| Mode B (Parallel) | 解空间探索 | 问题本身有多个合理解 |
| Mode C (Debate) | 论据博弈 | 双方分歧且有可检验依据 |

关键约束：Mode C 的 **3轮硬性封顶** 是防止陷入"修辞循环地狱"的工程保护。

### 核心设计张力

这个 skill 存在一个根本张力：**真正独立性的获取成本极高**。
- 传给 Codex 的 prompt 越详细，独立性越低（被锚定）
- 传的越少，Codex 的上下文理解越差（无效审查）

这需要在提示词工程层面精心设计"传什么、不传什么"。

## Codex (GPT-4o) 独立视角

### 核心理念：受控的异质性

基于描述，这个 skill 的核心理念不是"再找一个模型帮忙"，而是在高风险场景里引入一种"受控的异质性"。它试图解决的根本问题是：单一模型即使很强，也会被自己的推理路径、训练分布、奖励偏好和上下文 framing 锁住，导致"看起来非常合理但其实错了"的错误难以被自己发现。尤其在代码、技术方案和临界事实判断上，真正危险的不是不会，而是**"错得很顺"**。

从软件设计哲学看，这套设计很像高可靠系统里的 **N-version programming**、审计隔离和 defense in depth。它承认"主代理不可信但可用"，所以不把正确性寄托在主代理的自省上，而是寄托在"独立生成 + 交叉验证 + 必要时执行验证"上。

引入 Codex CLI 的价值也不只是"换个模型"，而是它能**执行命令、读环境、跑验证**，这让验证从纯语言比较，升级成了"语言判断 + 外部世界约束"。

### 三种模式的边界

应该明确区分为"成本、独立性、争议强度"三个维度：

- **Mode A / Review** 最适合默认使用。适用于代码修改、重构、脚本、安全相关操作、依赖环境的结论、以及任何"Claude 已经形成强叙事"的场景。成本最低，也最符合工程流。

- **Mode B / Parallel** 适合问题本身就存在多种合理解，且不希望第二个模型被第一个模型污染的情况，比如架构取舍、API 设计、技术选型、模糊需求解释。重点是保留真正独立的初始解空间。

- **Mode C / Debate** 不应常态化，适合高分歧、高代价、且双方都能提出可检验依据的问题。**只在"争议可收敛"时有意义**；如果问题本身没有外部裁判标准，辩论很容易退化成更长的幻觉生成。

### 发现的关键设计缺陷

**1. 触发条件太主观**
"回答异常流畅时"这个信号对，但太主观。应改为风险分级规则：高破坏性操作、涉及环境状态、事实可能过时、存在两个以上合理方案、无法本地执行快速验证、主模型置信度高但证据链短。

**2. Mode A 只审查结论，不够**
很多错误不是结果错，而是过程错（跳过验证、默认隐含前提、忽略边界条件）。Mode A 应让 Codex 审查：结论 + 关键假设 + 验证命令 + 未验证部分。

**3. 三种模式应建立升级路径，不应平铺**
合理工作流：先 Mode A，发现核心分歧来自方案空间则升 Mode B，仍有不可消解冲突则进入有限轮数的 Mode C，最后强制落到可执行验证或人工裁决。

**4. Mode C 需要"证据类型约束"**
语言代理会越来越擅长为已有立场找论据。应要求每轮发言标注：可执行验证、源码/文档证据、经验性推断、纯逻辑假设。没有新增证据的回合应直接终止。

**5. 多模型一致性被误判为正确性（最危险的陷阱）**
两个模型同意，并不代表正确，只代表共享了某些训练分布和工程常识。它提升的是发现分歧和打破单路径偏差的概率，不是提供真值。真正接近真值的，仍然是外部验证：运行代码、查文档、看环境、测结果。

**6. 缺少失败模式记录机制**
应记录：为什么触发、发现了什么问题、哪种模式最有效、哪些分歧最后被验证为谁对。长期看，这能把 skill 从"流程习惯"升级成"误差画像系统"。

## 综合结论

### 两模型共识

1. **"受控异质性"** 是正确方向——不是"多模型投票"（会强化共同偏差），而是"不同训练路径产生的独立视角"
2. 这个设计的本质是**认知制衡机制**，类似于 N-version programming 和 defense in depth
3. **最大风险不在模型能力，在流程的独立性是否被污染**

### Codex 独立发现（已纳入 v1.1 改进）

- **多模型一致性 ≠ 正确性** — 最危险的认知陷阱，需要在 SKILL.md 中明确标注
- **Mode A 应审查假设而不只是结论** — 已在 v1.1 中改进 prompt 模板
- **三种模式应是升级路径而非并列** — 已在 v1.1 中重新设计为 A→B→C 升级链
- **Mode C 需证据类型标注** — 已在 v1.1 中加入 `[可执行验证]` 等标注规范

### Claude 独立强调（未来迭代方向）

- **Prompt 设计的独立性悖论** — 需要更明确的"传什么/不传什么"规范
- **触发条件的主观性问题** — 需要改为可操作的风险分级规则（已在 v1.1 中改进）

## 对 v1.0 → v1.1 的具体改动

基于此次讨论，v1.1 的 SKILL.md 做了以下改进：

1. **触发判断**：从"感觉类"改为"风险分级类"（6个明确触发条件）
2. **三种模式**：重新定义为升级链路（A→B→C），不再是平行选择
3. **Mode A prompt**：增加"关键假设"和"未验证部分"审查
4. **Mode C**：加入证据类型标注系统（`[可执行验证]`等4种）
5. **终止规则**：明确"没有新证据则提前终止"，不强制跑满3轮
6. **注意事项**：明确"两模型一致 ≠ 正确"
