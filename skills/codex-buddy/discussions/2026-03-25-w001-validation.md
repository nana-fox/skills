**Codex：**

> `codex-buddy` 不在本会话的已安装 skill 列表里，所以我按仓库中的 SKILL.md、STATUS.md 和设计文档直接做独立判断。
>
> 1. 这轮里"自主执行规则"如果严格按当前文档定义，主要指的是 SKILL.md#L107 这一组行为，而不是更广义的"自主选题/自主迭代"。具体有 6 条：有决策但暂时没有验证路径时，停在 Mode B，不升级辩论；有可被实验或文档裁决的核心分歧时，升 Mode C 或直接验证；涉及不可逆操作时，不能以模型共识收尾；双方一致且下一步仍在原请求内、可逆、可验证时，直接执行，不再追问"要继续吗"；只有在超出原请求、缺关键输入、不可逆、有外部副作用时才停下问用户；因这些原因未执行时，必须把阻断原因写出来。如果只聚焦 W-001，本质上最该验证的是后 3 条，因为那才是"收敛后是否自主执行"。
>
> 2. 以当前 evals.json 的形态，不能验证 W-001。它现在测的是"是否触发"和"模式是否选对"，不是"收敛后到底执行了还是追问了"。W-001 要自动化，至少需要可回放的对话级/轨迹级 eval，能断言 4 类结果：`direct_execute`、`ask_missing_input`、`stop_destructive`、`stop_external_side_effect`。如果 eval 只能比对一句 `expected_output`，那它永远只能测"口头表态"，测不到"真实行为"。
>
> 3. 当前 `done_when` 写法本身就不对，因为它把验收条件和运行时状态混在了一起：`human_gate` 不是 done 条件，只是阻断状态。如果无法自动化，我建议把它改成可达的人工验收条件，例如：`done_when: "人工确认: 提供至少 2 段真实对话 transcript，其中 1 段显示一致+可逆+在请求范围内时代理直接执行，1 段显示命中边界条件时代理停止并说明 gate 原因"`。这样 `done_when` 仍然描述"完成长什么样"，而 `human_gate: REQUIRED:missing_input` 只在缺少这些 transcript 时作为阻断状态出现。这也和设计文档里"不能自动化就改成人工确认门"的原则一致。
>
> 4. 我对当前 skill 最大的设计缺陷的独立判断是：它仍然没有把"验证责任"做成一等公民，所以整套机制还可能退化成"再问一次另一个模型"。也就是，当前版本缺少一个强约束的验证契约，明确区分"哪些是已验证事实、哪些是假设、谁负责验证、什么时候必须升级到执行验证/人工验证"。这比 W-001 更根本，因为没有这层约束，自主执行只会把一个不够可审计的结论更快地执行出去。如果只允许我先修一个点，我会先恢复 `Verification Escalation Matrix + Output Contract`，再做 W-001 的 replayable eval。
