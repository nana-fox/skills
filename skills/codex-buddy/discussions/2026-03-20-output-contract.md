# 讨论：Output Contract 标准输出模板

**日期：** 2026-03-20 | **模式：** Mode A（Claude 先独立方案，Codex 独立审查） | **结果：** 快速收敛，Codex 额外发现"验证责任"结构性缺陷

---

## 话题

v1.5 的 CHANGELOG 中 Codex 指出：Mode A/B/C 各自的 Codex prompt 都没有规定返回格式，Codex 每次输出结构随机——有时是表格，有时是散文，有时是列表。Claude 接到输出后无法一致性地提取"分歧点"和"已验证结论"。本轮主题是为 Codex 输出定义标准格式（Output Contract），让整个协作流程可预期、可解读。

---

## 第一轮：各自开场

**Claude：**

> 当前问题：各 Mode 的 Codex prompt 没有格式约定，输出结构随调用和 prompt 措辞变化。
>
> 改动方案：在 `Prompt 独立性协议` 之后新增 `## Output Contract` 小节，定义 5 个字段：
> ```
> 独立结论 / 与 Claude 一致 / 分歧点 / 已验证 / 未验证
> ```
> 同时在 Mode A/B/C 的 bash template 末尾追加一行强制要求使用模板。
>
> 改动集中：只新增一个小节（约 10 行），三个 mode 的 prompt template 各加一行。

**Codex（原始输出）：**

> 建议改，不是只把 `Output Contract` 从一句原则变成一句更长的话，而是把它落成"强约束模板 + 提取规则 + 各模式 prompt 注入点"三件套。否则 Claude 还是会从自由文本里猜"哪些是事实、哪些已验证、哪些只是建议"。
>
> **1. 具体该怎么改**
>
> 先改这一句，让它从抽象原则变成指向明确章节：
>
> 改动前：`Output Contract（区分事实/假设/建议/未验证）`
> 改动后：`Output Contract（必须使用下文"标准输出模板"）`
>
> 然后新增一个独立章节，放在"Prompt 独立性协议"后，"收尾四问"前：
>
> ```text
> [结论]
> - <一句话主结论>
> - 置信度：high | medium | low
>
> [事实]
> - F1. <已由输入证据直接支持的事实>
>   - 来源：<代码/日志/报错/命令输出/文档>
>
> [已验证]
> - V1. <已经执行过的外部验证及结果>
>   - 验证方式：<命令/测试/运行/文档核对>
>   - 结果：<pass/fail/observed>
> - 若无，明确写：无
>
> [假设]
> - A1. <当前仍未验证但影响结论的假设>
> - 若无，明确写：无
>
> [建议]
> - R1. <建议动作>
>   - 依据：<对应 F#/V#/A#>
>
> [未验证]
> - U1. <本应验证但尚未验证的点>
>   - 建议验证：<最小外部验证动作>
> - 若无，明确写：无
>
> [与 Claude 对比]    # 仅 Mode A 使用
> - 一致：...
> - 分歧：
>   - D1. <分歧点>
>     - 类型：结论 | 假设 | 证据解释 | 验证优先级
>     - 我方依据：<F#/V#/A#>
>     - 需要的裁决验证：<最小验证动作>
> ```
>
> 约束：
> 1. `事实` 只能写输入中可直接观察到的内容，禁止掺入推断。
> 2. `已验证` 只能写已执行的外部验证；"我认为""通常""根据经验"不得放入本节。
> 3. `假设` 与 `未验证` 不得为空缺省，必须显式写 `无`。
> 4. `建议` 必须引用 `F#` / `V#` / `A#`，避免无依据建议。
> 5. Mode B/C 首轮禁止出现 `[与 Claude 对比]`。
> 6. 若没有 `已验证` 且结论依赖关键假设，`置信度` 不得为 `high`。
>
> 接着再补一个"Claude 读取规则"：
>
> ```md
> 综合 Codex 输出时，Claude 只做以下映射：
> - `已验证` → 视为当前最强证据
> - `事实` → 视为已观察但未必已外部验证
> - `假设` → 视为仍待裁决的前提
> - `未验证` → 直接进入 Verification Gap
> - `与 Claude 对比 / 分歧` → 直接进入 Key Divergence
> ```
>
> **2. 这个改动可能引入的新问题**
>
> - 模板过重。简单 review 也要填全字段，使用成本上升，大家会偷懒跳过。
> - 产生"格式化的错觉"。模型把格子都填满，看起来更严谨，但内容仍可能错。
> - `事实` 和 `已验证` 的边界容易被混淆，尤其是"看了代码"到底算观察还是验证。
> - `建议必须引用 F/V/A` 会逼模型做形式化引用，可能出现机械编号、降低可读性。
> - Mode C 本来偏辩论，套统一模板后可能削弱交锋感，输出会更像审计报告。
>
> 我会在文档里补一句缓冲条款：`模板外补充说明允许存在，但不得承载关键结论；关键结论必须落在模板字段内。`
>
> **3. 我最担心的设计缺陷（超出本轮主题）**
>
> 我最担心的不是 Output Contract，而是这个 skill 对"验证责任"定义得太虚。
>
> 现在文档写的是"真值来源：执行验证 > 文档 > 模型共识"和"可验证则必须执行验证"，但没有回答最关键的操作问题：
>
> - 谁来执行验证，Claude 还是 Codex？
> - 什么时候从 `read-only` 升到 `workspace-write`？
> - 如果验证成本高、环境不完整、权限不够，何时允许停在"未验证"而不是继续升级？
> - 哪类验证算"足够"，哪类只是又一层模型解释？
>
> 结果是：大家会很容易把这个 skill 用成"再问一次另一个模型"，而不是"组织一次真正的异质性校验 + 必要的外部验证"。
>
> 如果只允许我提一个结构级修复，我会加一个 `Verification Escalation Matrix`，把下面几类情况写死：
> - 纯事实核对：优先文档/命令验证，不够再求助 Codex
> - 代码 review：默认 `read-only`，出现关键分歧再跑测试/复现
> - 架构分歧：先 Mode B；只有存在可裁决实验时才升 C
> - 破坏性操作：没有外部验证方案时，不允许仅凭双模型共识执行
>
> 这比继续打磨 prompt 细节更重要，因为它直接决定这个 skill 最终是"第二意见工具"，还是"伪验证工具"。

## 共识与分歧

| 点 | Claude | Codex | 结论 |
|----|--------|-------|------|
| Output Contract 必须是"强约束模板 + prompt 注入" | 是（但只有 5 字段） | 是（7 字段 + 提取规则 + 三件套） | ✅ 采用 Codex 更完整的结构 |
| 新增 Claude 提取规则 | 未独立提出 | 明确提出，视为关键缺失 | ✅ 采纳，压缩成一行放在模板后 |
| 缓冲条款（模板外文本允许存在） | 未提出 | 明确建议 | ✅ 采纳，防止过度僵化 |
| 置信度约束（无验证不得 high） | 未提出 | 列为约束 6 | ✅ 采纳 |
| Verification Escalation Matrix | 未提出 | 认为是更根本的缺陷 | ➡️ 列入下轮 Agenda 首位，本轮不做 |

## Codex 超出本轮主题的发现（纳入下轮 Agenda 首位）

> 这个 skill 对"验证责任"定义得太虚——谁来执行验证、何时升级沙盒、何时允许停在"未验证"。
> 结果是大家会把这个 skill 用成"再问一次另一个模型"，而不是"真正的异质性校验"。
> 需要 `Verification Escalation Matrix` 写死各场景的验证要求。

**Claude 的判断：** Codex 的诊断准确。验证责任模糊是比 Output Contract 更根本的设计问题。本轮先落 Output Contract（有了结构化输出，才能清晰讨论验证缺口），下轮专攻验证责任矩阵。
