# 讨论：触发覆盖补全 + 收敛后自主执行规则

**日期：** 2026-03-23 | **模式：** Mode B × 2（两个独立问题各跑一轮） | **结果：** 收敛

---

## 话题

本轮聚焦两个独立但相关的设计漏洞：

1. **description 枚举漏洞**：当前 description 通过列举具体场景（code review、debugging、architecture）来描述 skill 的触发条件，但"讨论类"场景（workflow/strategy/iteration tradeoffs）被枚举遗漏，导致这类请求首轮不触发。由于 Red Flags 在触发后才读，漏触发无法被 body 内容补救——这是机制层面问题，不是执行层问题。

2. **收敛后行为缺失**：Mode B/C 综合后没有执行规则，代理完成讨论后仍然停下询问用户是否执行，形成设计漏洞——"综合结论"≠"确认检查点"，这是缺失规则导致的多余交互，不是谨慎执行的好习惯。

---

## 第一轮：各自开场

**Claude（独立分析）：**

> **问题 1：description 漏触发**
>
> 当前 description 是枚举式覆盖：列了 code review、debugging、architecture design 等场景。但枚举终究是枚举——遇到 "我想讨论这个迭代策略是否合理" 这类请求，没有关键词命中，skill 不触发。
>
> 修法：在 description 加入上位类 `consequential judgment`，并显式列出 `workflow / strategy / iteration tradeoffs` 作为子场景。上位类比枚举更稳定，不依赖请求措辞。
>
> **问题 2：收敛后行为**
>
> Mode B 流程的收敛节点后，当前 SKILL.md 没有写下一步。代理收敛后会习惯性问"要我执行吗？"——这个问题在大多数情况下是多余的，因为用户发起 codex-buddy 的目的就是要执行某件事，不是只看讨论记录。
>
> 修法：加 3 条规则——两边一致且下一步可逆/在范围内 → 直接执行；超出原请求/不可逆/外部副作用 → 必须问用户；阻断时写收尾摘要。

**Codex（独立 review，原始输出）：**

> **关于 description 是"唯一触发机制"这件事：**
>
> Claude 的修法方向是对的，但我要强调一个 Claude 没有明说的前提：description 是 skill 的唯一触发机制。Red Flags 和 body 里的任何内容，在 skill 没有触发的情况下根本不会被读到。这意味着 body 里的任何保护措施都无法补救首轮漏触发——这是机制层面问题，不是执行层问题。
>
> 所以修 description 不是"优化措辞"，而是"堵死唯一的单点故障"。优先级应该是 P0。
>
> 另一个 Claude 没有明说的推论：**宁可误触发，不要漏 consequential judgment**。cost 不对称：漏触发 = 用户根本得不到双模型验证；误触发 = 多走一遍流程但结果不变。所以 description 应偏保守，上位类覆盖宽一些比枚举精确但有漏洞要好。
>
> **关于收敛后行为：**
>
> Claude 的三条规则设计是合理的，但缺一个关键观察："综合结论"≠"确认检查点"。
>
> 现有流程把收敛理解为"我们讨论完了，接下来怎么做请你决定"——但这个理解是错的。Mode B 的设计目的是降低单模型错误率，不是把决策权退还给用户。如果收敛后还要问用户，那双模型验证的价值就折损了一半：用户本来信任两个模型一致才调用 skill，结果两个模型收敛后又被要求重新决策。
>
> 这是设计漏洞，不是执行习惯。补规则是正确的修法。
