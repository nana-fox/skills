# 讨论：Verification Escalation Matrix（验证责任矩阵）

**日期：** 2026-03-20 | **模式：** Mode A（Claude 先独立方案，Codex 独立审查） | **结果：** 快速收敛，Codex 额外发现 Mode A 默认值的锚定风险

---

## 话题

v1.6 中 Codex 指出：skill 对"验证责任"定义太虚——没有回答谁来执行验证、何时升级沙盒、何时允许停在未验证。结果是 skill 容易被用成"再问一次另一个模型"，而不是"真正的异质性校验"。本轮主题是补充 Verification Escalation Matrix，把各场景的验证责任写死。

---

## 第一轮：各自开场

**Claude：**

> 当前问题：Escalation Rule 写着"可验证则必须执行验证"，但没有给出可操作规则，用户面对"可验证"判断无从下手。
>
> 改动方案：在 `注意事项` 之前新增一个简洁的"验证升级矩阵"，用表格写死 4 类场景的验证责任：
>
> | 场景 | 谁验证 | 沙盒 | 何时可停留于"未验证" |
> |---|---|---|---|
> | 事实核对 | Claude 优先（查文档/命令） | read-only | 无法访问外部资源时 |
> | 代码审查 | Codex（read-only 扫描） | read-only | 分歧不影响结论时 |
> | 关键分歧 | 谁提出谁验证，优先运行测试 | workspace-write | 环境不完整时，明确标注[未验证] |
> | 破坏性操作 | 必须外部验证，不得仅凭模型共识 | workspace-write | **不允许** |
>
> 这张表放在正文里，精简为 10 行内，不需要长篇解释。

**Codex（原始输出）：**

> 我会把这次改动聚焦成一个判断：现在的 `codex-buddy` 把"是否找第二个模型"定义得比较清楚，但把"谁必须做外部验证、做到哪一步才允许停"定义得不够清楚。所以建议不要继续强化 mode 说明，而是把"验证责任"提升成与 Mode A/B/C 正交的第一层决策。
>
> **1. 具体该怎么改**
>
> 第一处，改"工作模式（升级链路）"的定位。现在它把"分歧处理"和"验证升级"混在一起了。
>
> 改动前：
> ```md
> ## 工作模式（升级链路）
> 先用 Mode A，发现分歧来自方案空间时升 B，仍有不可解冲突时升 C。
> ```
>
> 改动后：
> ```md
> ## 决策顺序
> 先定 **Verification Level（验证级别）**，再选 **Mode（协作模式）**。
> - Verification Level 决定：谁负责外部验证、是否允许停在未验证、何时从 `read-only` 升到 `workspace-write`
> - Mode 决定：Claude 与 Codex 如何交换观点（Review / Parallel / Debate）
> **顺序不可颠倒。** 先选 Mode 容易把 skill 用成"再问一次另一个模型"。
> ```
>
> 第二处，新增 Verification Escalation Matrix（V0-V3 完整矩阵）直接加在"工作模式"前面：
>
> | Level | 典型场景 | 默认责任人 | 默认沙盒 | 何时升级 | 何时允许停在未验证 |
> |------|------|------|------|------|------|
> | V0 | 低风险措辞改写、思路补充、非事实性建议 | 无需 Codex | read-only | 不升级 | 用户明确只要思路，不把结果当事实/决策依据 |
> | V1 | 可由文档核对的事实、API/CLI 用法、非破坏性代码理解 | Claude 或 Codex 任一方完成文档/源码核对 | read-only | 文档冲突或证据不足时升 V2 | 已明确列出缺失证据与风险，且结论不用于高风险操作 |
> | V2 | 本地可执行验证的结论：测试、重现 bug、性能数字、迁移脚本 dry-run | **提出结论的一方负责先验证**；另一方负责找遗漏和反例 | 先 read-only，需执行命令则升 `workspace-write` | 结论依赖未验证假设、双方分歧可通过执行裁决、用户将据此实施变更 | 仅当用户明确接受"先给分析不执行"，且输出中保留 `未验证` 与最小验证动作 |
> | V3 | 破坏性/不可逆操作：删除数据、生产变更、权限/安全边界修改 | **必须有人执行外部验证**；若 Codex 执行则需明确升级；否则 Claude 明确移交给人工 | `workspace-write` 或人工外部环境 | 默认不允许停在纯模型共识 | 仅允许输出"停止执行 + 缺失验证清单"，不允许给出可直接执行的肯定结论 |
>
> 责任规则（5条）：
> 1. 谁先提出会影响用户决策的主张，谁先承担最小验证责任。
> 2. Codex 不是默认执行者；只有当验证动作在项目工作区内可安全完成，才升到 `workspace-write`。
> 3. 若验证需要真实外部系统、生产权限、付费资源或不可回滚副作用，责任转交给人工，不得让 Codex 代执行。
> 4. 分歧若可被低成本外部验证裁决，不允许停在"双方各有道理"。
> 5. 高风险场景下，未验证本身就是结论的一部分：应输出"不能确认可执行"。
>
> 第三处，替换硬规则（五→六条），加入 `Verification First`。
>
> 第四处，给 Output Contract 增加验证相关字段：
> ```
> [验证级别] V0 / V1 / V2 / V3
> [验证责任] Claude / Codex / 双方 / 人工
> [升级决策] stay-read-only / escalate-codex-workspace-write / handoff-to-human
> ```
>
> 第五处，改"沙盒"注意事项从提示变成规则，绑定到 V2/V3 条件。
>
> **2. 这会引入什么新问题**
>
> - 复杂度上升，低风险场景下用户会嫌步骤重
> - 分类争议：很多任务横跨 V1/V2/V3（例如数据库迁移脚本 review）
> - `workspace-write` 可能被形式化滥用：只要判断是 V2 就以为"该升"
> - Output Contract 字段越多，模型越容易机械填写，字段形式对了但判断质量不一定提高
> - Mode C 本来偏辩论，套统一模板后可能削弱交锋感
>
> **3. 我最担心的现有设计缺陷（超出本轮主题）**
>
> 我最担心的不是"验证责任模糊"本身，而是 `Mode A — Review（默认）` 这个默认值。
>
> 即使你写了"先独立判断再对比"，只要 Claude 的结论已经进 prompt，Codex 的"独立性"就会被削弱。结果就是这个 skill 很可能长期运行成"受控的复述与补充"，而不是你想要的"异质性校验"。
>
> 如果只改一件事，我会把默认心智模型从：
> "先 Mode A，再看要不要验证"
> 改成：
> "先判定验证级别；只有在 V1 且目标是 review 时才默认用 Mode A；V2/V3 更应该先想谁去执行验证，而不是谁先评论谁。"
>
> 这比单纯补一句"可验证则必须验证"更关键，因为它直接改变这个 skill 被使用的方式。

## 对 SKILL.md 的改动（v1.7）

1. **工作模式入口**：加"先按验证矩阵确定验证级别，再选 Mode"提示
2. **新增 `## Verification Escalation Matrix`**：V0-V3 表格 + 3 条责任规则
3. **六条硬规则**：加入 `Verification First`
4. **Output Contract 模板**：顶部加 `[验证级别][验证责任][升级决策]` 三字段
5. **约束 ⑥**：V2/V3 且无 `已验证` 时 `升级决策` 不得为 `stay-read-only`
6. **沙盒说明**：从提示升级为规则，绑定 V2/V3 条件，V3 明确转交人工

SKILL.md 行数：149 → 169 行（+20 行）

## Codex 超出本轮主题的发现（纳入下轮 Agenda）

> Mode A 作为默认值天然带锚定风险。即使写了"先独立判断再对比"，只要 Claude 的结论已经进 prompt，Codex 的独立性就会被削弱。应该把默认心智模型从"先选 Mode A"改成"先判定验证级别，再选 Mode"——V2/V3 场景下应该先想谁去执行验证，而不是谁先评论谁。

**Claude 的判断：** Codex 的锚定风险诊断是准确的。Mode A 之所以是默认，是因为它最轻量，但"轻量"和"独立性"之间有内在张力。这个问题比增加一个矩阵更根本，需要单独讨论。本轮先落矩阵（有了级别定义，才能讨论 Mode 选择的合理性），下轮专攻 Mode A 锚定问题。
