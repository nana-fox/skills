# 讨论记录：Prompt 独立性规范

**日期：** 2026-03-20
**模式：** Mode B（Parallel — 两模型独立分析后综合）
**主题：** 传什么/不传什么给 Codex 才能真正保持独立性？

---

## 问题

codex-buddy 的核心价值来自"受控异质性"——两个模型独立判断。但现实问题是：
- 如果把太多 Claude 的上下文传给 Codex，Codex 会被锚定，失去独立性
- 如果传太少，Codex 缺乏项目背景，给出无用的泛泛建议

具体问题：
1. 哪些内容应该传？哪些绝对不能传？
2. 有没有结构化的 prompt 模板？
3. 当前 Mode A/B/C prompt 示例的独立性保护够吗？
4. 这个 skill 目前最大的设计漏洞是什么？

---

## Claude 独立视角（先写，未看 Codex 答案）

核心矛盾：传太多 → Codex 被锚定；传太少 → Codex 无法给出有效判断。

**应该传的：**
- 问题的原始描述（用户的原话，不加 Claude 的解读）
- 项目技术栈（让 Codex 理解约束）
- 需要审查的产物本身（代码/结论/方案）

**不应该传的：**
- Claude 的推理过程（最危险——会让 Codex 顺着同一条推理链走）
- Claude 的结论摘要（避免 Codex 变成"找反例"而不是独立判断）
- 对问题的定性词（"这个方案好/不好"）

**我识别的核心张力：**
传给 Codex 的信息需要"中性背景"和"问题事实"，但要严格排除"Claude 的解读框架"。技术栈、系统边界这类客观约束是安全的，不会锚定。

## Codex (GPT-4o) 独立视角（完整原文）

> 这个 skill 现在最需要补强的不是"再写几句强调独立性"，而是把"允许传入的上下文"做成硬规则。否则调用者很容易一边口头上说 independent，一边把 Claude 的结论、措辞、优先级、风险判断整包塞给 Codex，结果形式上是双模型，实际上还是单模型放大回声。

### Codex 的四层分类

**层 1 — 可传：任务事实**
- 用户原始问题、目标、约束、成功标准
- 可观察环境事实：仓库路径、文件名、报错原文、命令输出、接口定义、测试结果
- 需要审阅的原始材料本身

**层 2 — 谨慎传：最小项目背景**
- 只传 Codex 做判断必需的局部上下文
- 不传整段项目历史、团队偏好、你已经倾向的方案

**层 3 — Review 模式可传：Claude 产出，但要降锚**
- 必须明确字段分隔：原始任务 / 客观证据 / Claude 的观点（待审）
- 必须要求 Codex "先独立重建判断，再对比 Claude"

**层 4 — 绝对不要传**
- "Claude 认为方案 A 更好，因为……你帮我确认"
- "我们已经基本确定这样做对"
- Claude 对 Codex 的预设角色，如"请扮演反方"
- 任何会把 Codex 锚定到结论/立场/语气/优先级排序的 framing

**一句话原则：`传证据，不传结论；传约束，不传倾向；传问题，不传答案。`**

### Codex 指出的当前 prompt 具体问题

**Mode A 问题：**
- 把"Claude 的产出"整块塞进去，没有要求 Codex 先独立重建问题
- 没有要求附上原始任务和客观证据
- "与你的独立判断有哪些分歧"这句话在前面已经被强锚定了，问也白问

**Mode B 问题：**
- `请独立回答：<原始问题>` 太弱
- 没有规定应该提供哪些上下文，也没有禁止传入锚定信息
- 没有要求 Codex 区分"事实、假设、缺失验证"，输出可能流畅但无法对比

**Mode C 问题（最有价值的诊断）：**
- 示例里"Claude 的立场是：<Claude 的观点>"会让第 1 轮变成被动反驳
- 现在的 Mode C 实际上是"**反方审稿模式**"，不是真正独立 Debate
- 正确做法：C1 双方先各自独立陈述立场和证据类型，C2 再交换反驳

### Codex 提出的五条硬性协议

```
Context Order: 原始问题 > 客观证据 > 最小背景 > Claude 观点
Context Budget: Claude 观点不能超过总上下文的 25%
Escalation Rule: 若分歧可通过命令/测试验证，必须升到执行验证
Output Contract: Codex 必须显式标出：事实 / 假设 / 建议 / 未验证
Abort Rule: 如果没有足够原始证据，先返回"证据不足"，不给结论
```

> 如果你后续要改这个 SKILL.md，我建议优先加一节：`Prompt Independence Contract`。这会比继续扩写 Mode 示例更有价值。

## 综合结论

### 两模型共识
- **传证据不传结论** 是核心原则（两者独立得出相同原则）
- Mode A 必须要求 Codex "先独立重建判断，再对比 Claude"
- Mode B 当前模板太弱，需要结构化字段

### Codex 明显强于 Claude 的地方（完全采纳）

| Codex 的贡献 | 采纳情况 |
|-------------|---------|
| 四层分类（比 Claude 的两层更可操作） | ✅ 采纳 |
| Mode C 诊断：现在是"反方审稿"不是 Debate | ✅ 采纳，重设计 C1/C2 |
| Abort Rule：证据不足时先返回"证据不足" | ✅ 采纳（Claude 完全没想到） |
| Context Budget: Claude 观点不超过 25% | ✅ 采纳（量化边界很实用） |
| `Prompt Independence Contract` 作为独立章节 | ✅ 采纳，新增章节 |

### Claude 补充 Codex 未说明的
- 技术栈/系统边界等"中性客观约束"其实是安全的，不会锚定，Codex 的"谨慎传"分类可以更明确

## 对 SKILL.md 的具体改动（v1.1 → v1.2）

1. **新增章节：`Prompt 独立性协议`**（四层规则 + 五条硬性协议）
2. **Mode A prompt 重写**：字段隔离结构 + "先独立重建判断"要求 + Output Contract
3. **Mode B prompt 重写**：完整结构化模板（任务/硬约束/证据/输出要求）
4. **Mode C 重设计**：C1（双方独立陈述）→ C2（交换反驳），硬性上限改为2轮
5. **终止规则加强**：无新可验证证据时立即终止
